Operator Hierachy¶

The classes are nested as follows:

nesting of the operator classes: solid lines means “has many”, dashed lines means “evolves into”¶

  • OperatorGrid

    • this is the master class which administrates all evolution kernel operator tasks

    • it is instantiated once for each run

    • it divides the given range of \(Q^2\) into the necessary threshold crossings and creates a OperatorMaster for each

    • it recollects all necessary operators in the end to create the PhysicalOperator following the instructions of FlavourTarget

  • OperatorMaster

    • this represents a configuration for a fixed number of flavours

    • it creates an Operator for each final scale \(Q^2\)

  • Operator

    • this represents a configuration for a fixed final scale \(Q^2\)

    • this class is only used internally

    • its OpMember are only valid in the current threshold area

  • PhysicalOperator

    • this is the exposed equivalent of Operator, i.e. it also lives at at fixed final scale

    • its OpMember are valid from the starting scale to the final scale

  • OpMember

    • this represents a single evolution kernel operator

    • inside Operator they are in “raw” evolution basis, i.e. \(\tilde{\mathbf{E}}_S, \tilde{E}_{ns}^{\pm,v}\), and they never cross a threshold

    • inside PhysicalOperator they are in “true” evolution basis, i.e. they evolve e.g. \(\tilde V, \tilde T_3\) etc., so they are eventually a product of the “raw” basis (see Matching Conditions)
